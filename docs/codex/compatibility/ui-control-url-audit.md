# UI ↔ PI Control URL Audit

## Audit Scope

This audit lists the exact PI/control URLs generated by the React UI after aligning with the confirmed PI control API. It is based on code inspection of `src/global.ts`, `src/repositorys/ProductionRepository.ts`, `src/epics/productionEpics.ts`, `src/enums/eConfirmStates.ts`, `src/repositorys/ProductionRepository.test.ts`, and imported PI control context in `docs/codex/external/control-context.md`.

## URL Constants Used by the UI

| Constant | Exact value | Notes |
|---|---:|---|
| `BaseURL` | `http://192.168.178.37:5000/` | Status, recipe, temperature, water status, availability, and `/next`. |
| `CommandsURL` | `http://192.168.178.37:5000/Command/` | Command-style POST URLs. |
| `ConfirmURL` | `http://192.168.178.37:5000/Confirm/` | Concrete confirmation POST URLs. |

The socket.io URL is derived as `BaseURL.replace(/^http/, 'ws')`, which produces `ws://192.168.178.37:5000/`.

## Confirmed PI Control API Baseline

- `GET /Available/` is the UI-facing availability endpoint.
- `GET /WaterStatus` and `GET /WaterStatus/` are both supported and return `{ liters, openClose }` from startup.
- `GET /Status/` returns structured PI control status.
- `POST /Recipe/` accepts recipe JSON and returns `201` on success.
- `POST /Command/TurnOn` and `POST /Command/TurnOff` are supported no-value command aliases.
- Existing value-bearing command syntax remains supported, including `StartBrewing:""` and `AgitatorInterval:""`.
- `Wait` is a status, not a confirmation command. The UI must not call `POST /Confirm/Wait`.
- Valid confirmation endpoints are `Iodine`, `Mashup`, `Cooking`, `Boiling`, and `Decoction`.

## Calls Explicitly Prevented

| Invalid call | UI behavior |
|---|---|
| `POST http://192.168.178.37:5000/Confirm/Wait` | Not generated. `Wait`/unknown/generic waiting states may display status text, but confirmation action is disabled/no-op unless a concrete confirmation type is known. |

## Final Verification Addendum

The final confirmed endpoint contract is authoritative in `docs/codex/compatibility/final-ui-control-compatibility-report.md`.

Final notes:

- `GET /` is preserved by PI control but is not used by the UI for availability.
- `GET /Available/` is the UI-facing availability endpoint.
- `GET /WaterStatus` and `GET /WaterStatus/` are both supported by PI control; the UI currently generates `GET /WaterStatus`.
- `POST /Command/TurnOn:""` and `POST /Command/TurnOff:""` remain preserved by PI control for compatibility; the UI intentionally generates no-value `POST /Command/TurnOn` and `POST /Command/TurnOff`.
- `POST /Confirm/Wait` is rejected by PI control with a controlled error and is not generated by the UI.
